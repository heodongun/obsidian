---# [혼공컴] 명령어

> 명령어는 실질적으로 컴퓨터를 작동시키는, 어쩌면 데이터보다 더 중요한 정보입니다. 명령어란 무엇이고, 어떻게 생겼으며, 그리고 컴퓨터를 어떻게 작동시키는지 함께 알아봅시다.

결론부터 말하자면 모든 소스코드는 컴퓨터 내부에서 명령어로 변환됩니다.
<br><br>
## 고급언어와 저급언어
고급언어 : 사람을 위한 언어
저급언어 : 명령어로 이루어져있습니다.
그래서 고급언어로 작성된 소스코드가 실행되려면 **반드시** 저급언어, 즉 명령어로 변환되어야합니다.

기계어란 0과 1의 명령어 비트로 이루어진 언어입니다.
기계어는 오로지 컴퓨터만을 위해 만들어진 언어이기 때문에 사람이 읽으면 그 의미를 이해하기 어렵습니다.
그래서 등장한 저급언어가 **어셈블리어** 입니다.

![](https://velog.velcdn.com/images/pobi/post/80a7a531-7d81-4776-91de-f4b8ecb15c2e/image.png)
위 어셈블리어 한줄한줄이 명령어라는 사실만 기억해 주세요.

프로그래밍 언어를 배워 봤따면 어셈블리어는 우리가 아는 언어들과는 사뭇 다르게 생겼다는 사실을 아실 겁니다. 어셈블리어는 0과 1로 이루어진 기계어를 읽기 편하게 만든 저급 언어일 뿐이므로, 개발자가 어셈블리어를 이용해 복잡한 프로그램을 만들기란 쉽지 않습니다.

"이러면 굳이 저급언어를 알아야하나요?"라는 질문을 하실수도 있습니다.
개발자, 게임 개발자, 정보 보안 분야등의 개발자는 어셈블리어를 많이 이용합니다.

어셈블리어를 읽으면 컴퓨터가 프로그램을 어떤 과정으로 실행하는지, 가장 근본적인 단계에서부터 하나하나 추적하고 관찰할 수 있기때문입니다.
<br><br>
## 컴파일 언어와 인터프리터 언어
그러면 고급언어는 어떻게 저급 언어로 변환될까요?
여기에는 크게 2가지, **컴파일 방식**과 **인터프리트 방식**이 있습니다.
![](https://velog.velcdn.com/images/pobi/post/9eefd235-ea37-4ce6-a82a-f3c4720af292/image.png)
컴파일 방식으로 작동하는 프로그래밍 언어를 **컴파일 언어**, **인터프리트** 방식으로 작동하는 프로그래밍 언어를 **인터프리터 언어**라고 합니다.

<br><br>
### 컴파일 언어
**컴파일 언어**는 컴파일러에 의해 소스코드 전체가 저급언어로 변환되어 실행되는 고급언어입니다.
대표적인 컴파일 언어로는 C언어가 있습니다.
컴파일 언어로 작성된 소스 코드는 코드 전체가 저급 언어로 변환되는 과정을 거칩니다. 이 과정을 **컴파일**이라고 합니다.
컴파일러를 통해 저급언어로 변환된 코드를 **목적코드**라고 합니다.
<br><br>
### 인터프리터 언어
인터프리터 언어는 인터프리터에 의해 소스 코드가 한 줄씩 실행되는 고급언어입니다.
소스코드를 한줄씩 저급언어로 변환하여 실행해주는 도구를 **인터프리터**라고 합니다.<br><br>

### 목적파일 vs 샐행파일
이미지로 이루어진 파일을 이미지 파일이라 부르고, 텍스트로 이루어진 파일을 텍스트 파일이라고 부르듯이 **목적코드**로 이루어진 파일을 **목적 파일**이라고 합니다.
목적 코드가 실행 파일이 되기 위해서는 **링킹**이라는 작업을 거쳐야 합니다.

<br><br>
## 핵심키워드
1. **고급언어**는 사람이 이해하고 작성하기 쉽게 만들어진 언어입니다.

2. **저급언어**는 컴퓨터가 직접 이해하고 실행할수있는 언어입니다.

3. 저급 언어는 0과 1로 이루어진 명령어로 구성된 기계어와 기계어를 사람이 읽기 편한 형태로 번역한 **어셈블리어**가 있습니다.

4. **컴파일언어**는 컴파일러에 의해 소스 코드전체가 저급 언어로 변환되어 실행되는 언어입니다.

5. **인터프리터 언어**는 인터프리터에 의해 소스 코드가 한 줄씩 저급 언어로 변환되어 실행되는 언어입니다.
<br><br>
## 명령어의 구조
명령어를 자세히 들여다보면 연산코드, 오퍼랜드, 주소 지정 방식이라는 개념을 알아보겠습니다.
<br><br>
### 연산코드와 오퍼랜드
명령어는 "무엇을 대상으로, 어떤 작동을 수행하라'는 구조로 되어있습니다.
![](https://velog.velcdn.com/images/pobi/post/f103d4be-765a-42b8-855e-b8b0a53d1a83/image.png)
명령어는 **연산코드**와 **오퍼랜드**로 구성되어있습니다.
'**명령어가 수행할 연산**'을 **연산코드**라 하고, '**연산에 사용할 데이터**'또는 '**연산에 사용할 데이터가 저장된 위치**'를 **오퍼랜드**라고 합니다. 연산코드는 연산자, 오퍼랜드 피연산자라고도 부릅니다.

연산코드가 담기는 영역을 연산 **코드 필드**
오퍼랜드가 담기는 영역을 **오퍼랜드 필드**라고 합니다.
<br><br>
### 오퍼랜드
오퍼랜드 필드에는 숫자와 문자 등을 나타내는 데이터 또는 메모리나 레지스터 주소가 올수있습니다.
즉 메모리 주소나 레지스터 이름이 담깁니다. 그래서 오퍼랜드 필드를 **주소필드**라고 부르기도 하죠.

여기서 오퍼랜드가 하나도 없는 명령어를 0 - 주소 명령어,
오퍼랜드가 하나인 명령어를 1 - 주소 명령어,
두개인 명령어를 2 - 주소 명령어,
세개인 명령어를 3 - 주소 명령어라고 합니다.
<br><br>
### 연산코드
연산코드는 크게 4가지 유형으로 나눌수있습니다.
1. 데이터전송
2. 산술/논리 연산
3. 제어 흐름 변경
4. 입출력 제어
네 그냥 있다고요 알아서 찾아봅시다.
<br><br>
### 주소지정 방식
"근데 굳이 데이터를 그냥 쓰면 되지 왜 굳이 주소를?"이라고 생각할수있다.

이는 **명령어 길이** 때문입니다.
오퍼랜드 필드 안에 메모리 주소가 담긴다면 표현할 수 있는 데이터의 크기는 하나의 메모리 주소에 저장할 수 있는 공간만큼 커집니다.

연산의 대상이 되는 데이터가 저장된 위치를 **유효 주소**라고합니다.

오퍼랜드 필드에 데이터가 저장된 위치를 명시할 때 연산에 사용할 데이터 위치를 찾는 방법을 주소지정방식 입니다.
다시 말해, 주소지정 방식은 유효 주소를 찾는 방법입니다.
<br>
### 즉시 주소 지정 방식
즉시 주소 지정방식은 연산에 사용할 데이터를 오퍼랜드 필드에 직접 명시하는 방법입니다. 빠릅니다.
<br>
### 직접 주소 지정 방식
직접 주소 지정방식은 오퍼랜드 필드에 유효 주소를 직접적으로 명시하는 방식입니다.
표현할수있는 오퍼랜드 필드의 길이가 연산 코드의 길이만큼 짧아져 표현할수있는 유효 주소에 제한이 생길수있습니다.
<br>
### 간접 주소 지정 방식
간접 주소 지정 방식은 유효 주소의 주소를 오퍼랜드 필드에 명시합니다.
다만 두번의 메모리 접근이 필요하기 때문에 앞서 설명한 주소 지정 방식들보다 일반적으로 느린 방식입니다.
<br>
### 레지스터 주소 지정 방식
레지스터 주소 지정 방식은 직접 주소 지정 방식과 비슷하게 연산에 사용할 데이터를 저장한 레지스터를 오퍼랜드 필드에 직접 명시하는 방법입니다.
다만 표현할수있는 레지스터 크기에 제한이 생길 수 있다는 점입니다.
<br>
### 레지스터 간접 주소 지정방식

레지스터 간접 주소 지정 방식은 연산에 사용할 데이터를 메모리에 저장하고 그 주소(유효 주소)를 저장한 레지스터를 오퍼랜드 필드에 명시하는 방법입니다.
<br><br>

## 대충 요약
1. 명령어는 연산 코드와 오퍼랜드로 구성됩니다.
2. 연산 코드는 명령어가 수행할 연산을 의미합니다.
3. 오퍼랜드는 연산에 사용할 데이터 또는 연산에 사용할 데이터가 저장된 위치를 의미합니다.
4. 주소 지정 방식은 연산에 사용할 데이터 위치를 찾는 방법입니다.
 - 즉시 주소 지정 방식 : 연산에 사용할 데이터
 - 직접 주소 지정 방식 : 유효 주소(메모리 주소)
 - 간접 주소 지정 방식 : 유효 주소의 주소
 - 레지스터 주소 지정 방식 : 유효 주소(레지스터 이름)
 - 레지스터 간접 주소 지정 방식 : 유효 주소를 저장한 레지스터~~
