# 2. common

> common 패키지는 Spring Boot 프로젝트에서 공통적으로 사용되는 코드나 로직을 모아 관리하는 공간입니다. 이 패키지에는 주로 상수, 유틸리티 함수, 공통 응답 객체, 예외 처리 로직 등을 포함시킵니다. 이러한 공통 요소들을 한곳에 모아두면, 코드 중복을 줄이고 프로젝트의 일관성을 유지할 수 있으며, 나아가 유지보수성도 크게 향상됩니다.<br>
예를 들어, API 응답에 사용되는 공통 응답 코드와 메시지를 common 패키지에 정의하면, 여러 클래스나 패키지에서 동일한 코드를 재사용할 수 있어 관리가 편리해집니다. 또한, 프로젝트에서 자주 사용하는 유틸리티 함수나 필터, 인터셉터 등을 이 패키지에 두면, 코드의 중복을 최소화하면서 공통 기능을 쉽게 활용할 수 있게 됩니다.<br>
따라서 common 패키지는 프로젝트 전반에서 반복적으로 사용되는 로직을 한데 모아, 개발 생산성을 높이고 코드의 유지보수를 용이하게 하는 중요한 역할을 합니다.


common 패키지는 프로젝트 내에서 자주 사용되는 공통 기능이나 유틸리티를 관리하는 패키지이다. 프로젝트 전체에서 재사용 가능한 코드, 예를 들어 유틸리티 클래스, 상수, 공통 예외 처리 등의 기능을 여기에 넣을수있다.

유틸리티 프로젝트란 프로젝트 내에서 공통으로 사용하는 기능(예: 문자열 처리, 날짜 변환 등)을 제공하는 클래스.


**common 패키지를 사용함으로써, 중복 코드를 줄이고 유지 보수성을 높일 수 있습니다.**
<br><br>
# 3. config
> config 패키지는 Spring Boot 프로젝트에서 애플리케이션의 설정과 구성을 관리하는 공간입니다. 이 패키지에는 보통 보안 설정, 데이터베이스 연결, CORS 정책 설정 등 애플리케이션의 동작 방식을 정의하는 클래스들이 포함됩니다. 이러한 설정을 config 패키지에서 중앙화해 관리하면, 애플리케이션의 확장성과 유지보수성이 크게 향상됩니다.<br>
예를 들어, 보안 설정을 관리하는 WebSecurityConfig 클래스를 config 패키지에 정의하여, Spring Security를 통해 API 접근 권한이나 세션 관리 방식 등을 쉽게 설정할 수 있습니다. 마찬가지로, CORS 설정이나 데이터베이스 연결 설정도 이 패키지에 모아두면, 전체적인 프로젝트 설정을 일관되게 유지하고 중앙에서 관리할 수 있게 됩니다.<br>
결론적으로, config 패키지는 프로젝트의 핵심 설정들을 한 곳에서 관리함으로써, 프로젝트의 일관성과 확장성을 유지하고, 필요할 때마다 다양한 설정을 빠르게 수정할 수 있게 해주는 중요한 역할을 합니다.

config 패키지는 **애플리케이션의 설정(configuration)**을 관리하는 패키지로, 주로 데이터베이스 설정, 보안 설정, 애플리케이션 전반에 걸친 주요 설정을 담당합니다. Spring Boot 애플리케이션에서는 이 패키지에 환경 설정 파일, Bean 설정, 스프링 관련 설정 클래스들을 위치시켜서 애플리케이션의 동작 방식을 제어한다.

주로 @Configuration 에노테이션을 사용한 클래스를 정의하며, 이를 통해 데이터베이스 연결 설정, 인증, 외부 API 설정 등을 처리한다.



# 4. controller

> controller 패키지는 Spring Boot 프로젝트에서 사용자 요청을 처리하고 응답을 반환하는 역할을 담당하는 공간입니다. 이 패키지의 클래스들은 주로 HTTP 요청을 받아들여, 이를 처리하고 적절한 데이터를 클라이언트에게 응답하는 기능을 수행합니다. MVC(Model-View-Controller) 패턴에서 Controller에 해당하며, 웹 애플리케이션의 진입점 역할을 합니다.<br>
예를 들어, 사용자가 브라우저에서 /api/v1/users 경로로 GET 요청을 보낸다면, 해당 경로에 매핑된 Controller 메서드가 요청을 처리하고, 사용자 목록을 조회한 후 이를 응답으로 반환합니다. 또한, POST, PUT, DELETE 등 다양한 HTTP 메서드를 사용해 데이터를 등록, 수정, 삭제하는 작업도 이 패키지에서 처리됩니다.<br>
결론적으로, controller 패키지는 사용자와 서버 간의 상호작용을 관리하며, 요청을 받아 필요한 비즈니스 로직을 실행한 후 적절한 응답을 반환하는 중요한 역할을 합니다. 이를 통해 애플리케이션의 흐름을 제어하고 사용자 요청을 효과적으로 처리할 수 있게 됩니다.

controller 패키지는 우리가 사랑하는 MVC 패턴에서 Controller에 해당하는 부분을 담당한다. MVC 패턴(Model-View-Controller)은 웹 애플리케이션의 구조를 체계적으로 나누기 위한 중요한 디자인 패턴으로, 다양한 애플리케이션에서 사용된다. MVC 패턴에 대해서는 다른 글에서 자세히 설명할 예정이지만, 간단히 설명하자면, Controller는 사용자로부터의 요청을 받아 처리하는 역할을 한다.
<br><br>
# 5. dto
>dto 패키지는 Spring Boot 프로젝트에서 **데이터 전송 객체(Data Transfer Object)**를 관리하는 공간입니다. DTO는 클라이언트와 서버 간에 데이터를 전송할 때 사용되며, 주로 API 요청 및 응답의 데이터 구조를 정의합니다. 이를 통해 데이터의 명확성과 일관성을 유지하고, 필요한 정보만을 효율적으로 전송할 수 있습니다.<br>
예를 들어, 사용자 등록을 위한 요청 데이터나 게시글 상세 정보를 포함하는 DTO 클래스를 정의함으로써, 클라이언트가 서버에 전송하는 데이터의 형태를 명확히 할 수 있습니다. DTO는 데이터의 유효성을 검증하고, 필수적인 정보만 포함시킬 수 있어 불필요한 데이터 전송을 방지하고 성능을 개선하는 데 기여합니다.<br>
결론적으로, dto 패키지는 클라이언트와 서버 간의 데이터 전송을 효율적으로 처리하기 위한 데이터 구조를 정의하고 관리하는 중요한 역할을 하며, 애플리케이션의 유지보수성과 확장성을 높이는 데 기여합니다.

dto(Data Transfer Object) 패키지는 클라이언트와 서버 간의 데이터 전송을 담당하는 객체들을 관리하는 패키지이다. 주로 **클라이언트 요청(request)**이나 **서버 응답(response)**에서 전송할 데이터를 정의한 클래스를 포함하고 있으며, 이 패키지 안에서 request와 response로 세분화될 수 있다.
<br><br>

# 6. entity
>Entity 패키지는 Spring Boot 프로젝트에서 데이터베이스의 테이블과 매핑되는 객체를 관리하는 공간입니다. 이 패키지에 포함된 클래스들은 주로 **JPA (Java Persistence API)**를 사용하여 데이터베이스와 상호작용하며, 애플리케이션의 데이터 구조를 정의합니다. 각 엔티티 클래스는 데이터베이스 테이블의 구조를 반영하고, 객체 지향적으로 데이터를 처리할 수 있는 기반을 제공합니다.

entity는 jpa발사대이다.
<br><br>
# 7. exception
>exception 패키지는 Spring Boot 프로젝트에서 예외 처리와 관련된 클래스를 관리하는 공간입니다. 이 패키지는 애플리케이션 실행 중 발생할 수 있는 다양한 예외 상황을 처리하기 위한 로직을 포함하고 있으며, 애플리케이션의 안정성과 사용자 경험을 향상시키는 데 중요한 역할을 합니다.<br>
예를 들어, 클라이언트가 잘못된 요청을 보내거나 데이터베이스에서 원하는 정보를 찾지 못했을 때, 해당 상황에 대해 적절한 예외를 정의하고 처리하는 것이 필요합니다. exception 패키지에서는 사용자 정의 예외 클래스나 특정 예외 발생 시 응답할 메시지를 포함하는 예외 처리기를 정의할 수 있습니다. 이를 통해 개발자는 예외 발생 시 일관된 방식으로 오류를 관리하고, 클라이언트에게 명확한 피드백을 제공할 수 있습니다.<br>
예외 처리는 애플리케이션의 안정성을 높이고, 사용자에게 보다 친숙한 경험을 제공하며, 코드의 유지보수성을 증가시키는 중요한 역할을 합니다. 예를 들어, 특정 조건이 충족되지 않았을 때 사용자에게 명확한 오류 메시지를 전달하면, 사용자는 문제를 더 쉽게 이해하고 해결할 수 있습니다.
<br>결론적으로, exception 패키지는 예외 발생 시 일관되게 오류를 관리하고, 사용자에게 명확한 피드백을 제공하여 애플리케이션의 안정성과 사용자 경험을 향상시키는 중요한 기능을 수행합니다.

exception은 예외처리 패키지이다.
<br><br>
# 8. filter
>filter는 Spring Boot 프로젝트에서 HTTP 요청과 응답을 가로채어 처리하는 중간 단계의 컴포넌트입니다. 필터는 요청이 컨트롤러에 도달하기 전에 또는 응답이 클라이언트에게 전달되기 전에 추가적인 처리를 수행할 수 있는 기능을 제공합니다. 이를 통해 보안, 로깅, 요청 수정, 응답 변환 등의 다양한 작업을 수행할 수 있습니다.<br>
예를 들어, 인증 필터는 클라이언트의 요청에서 JWT 토큰을 추출하여 인증을 수행할 수 있습니다. 만약 토큰이 유효하지 않다면 요청을 차단하고 적절한 오류 메시지를 반환할 수 있습니다. 이렇게 필터를 사용하면 요청이 컨트롤러에 도달하기 전에 필요한 모든 검증을 수행하여, 비즈니스 로직이 실행되기 전에 올바른 상태인지 확인할 수 있습니다.<br>
필터는 여러 개가 체인 형태로 연결되어 실행될 수 있으며, 각 필터는 순차적으로 요청과 응답을 처리합니다. 따라서, 필터를 적절히 활용하면 코드의 재사용성과 유지보수성을 높일 수 있습니다.<br>
결론적으로, filter는 HTTP 요청과 응답을 가로채어 추가적인 처리를 수행할 수 있는 컴포넌트로, 보안, 로깅, 데이터 변환 등 다양한 기능을 구현하는 데 사용되며, 애플리케이션의 전반적인 품질과 안정성을 높이는 데 기여합니다.

filter패키지는 요청을 미리 처리하여 요청이 유효한지 판단해준다.
이 필터는 여러개이다.
<br><br>
# 9. provider
>provider 패키지는 Spring Boot 프로젝트에서 서비스나 기능을 제공하는 컴포넌트를 관리하는 공간입니다. 이 패키지는 주로 특정 기능의 구현을 담당하는 클래스나 인터페이스를 포함하며, 외부 API와의 통신, 데이터 생성 및 변환, 인증 및 인가 등의 역할을 수행합니다.<br>
예를 들어, JWT(JSON Web Token) 기반 인증을 처리하는 JwtProvider 클래스는 클라이언트의 요청에 대해 토큰을 생성하고 검증하는 기능을 제공합니다. 이 클래스는 주로 암호화된 정보를 안전하게 생성하고 검증하기 위한 로직을 포함하며, 사용자의 인증 상태를 관리하는 데 중요한 역할을 합니다.<br>
provider 패키지는 일반적으로 비즈니스 로직과 데이터 접근 로직을 분리하여, 코드의 응집성을 높이고 유지보수성을 향상시키는 데 기여합니다. 또한, 의존성 주입을 통해 다른 컴포넌트와의 결합도를 낮추고, 각 기능을 독립적으로 테스트할 수 있는 장점을 제공합니다.<br>
결론적으로, provider 패키지는 특정 기능을 제공하는 컴포넌트를 관리하는 공간으로, 애플리케이션의 핵심 로직을 구현하고 유지보수성을 높이며, 기능의 재사용성을 증가시키는 중요한 역할을 수행합니다.

provider 패키지는 애플리케이션의 구성 요소나 서비스를 제공하는 클래스를 포함하는 패키지로, config 패키지와 비슷한 역할을 하면서도 약간의 차이점이 있다. config패키지는 프로젝트 전체라면 이 provider은 기능이나 서비스에 특화되어있다.

# 10. repository
>repository 패키지는 Spring Boot 프로젝트에서 데이터 저장 및 접근을 관리하는 공간입니다. 이 패키지는 주로 데이터베이스와의 상호작용을 담당하는 인터페이스와 클래스들을 포함하며, CRUD(Create, Read, Update, Delete) 작업을 수행하는 기능을 제공합니다.<br>
예를 들어, UserRepository라는 인터페이스는 사용자 관련 데이터에 대한 CRUD 작업을 정의할 수 있습니다. 이 인터페이스는 Spring Data JPA와 함께 사용되어, 기본적인 데이터 접근 메소드를 자동으로 생성하며, 개발자는 복잡한 SQL 쿼리를 작성할 필요 없이 데이터베이스와의 상호작용을 쉽게 처리할 수 있습니다.<br>
repository 패키지를 사용하면 데이터 접근 로직을 비즈니스 로직과 분리할 수 있어 코드의 응집성을 높이고, 유지보수성을 향상시키는 데 기여합니다. 또한, 데이터베이스에 대한 접근 방식을 통합함으로써, 코드의 일관성을 유지하고, 나중에 데이터베이스 구현을 변경할 때도 최소한의 수정으로 처리할 수 있는 유연성을 제공합니다.<br>
결론적으로, repository 패키지는 데이터 저장 및 접근을 관리하는 공간으로, 데이터베이스와의 상호작용을 효율적으로 처리하고, 코드의 유지보수성과 재사용성을 높이는 중요한 역할을 수행합니다.

config패키지가 데이터베이스와의 연결이라면 repository는 데이터베이스의 활용이다.

# 11. service
> service 패키지는 Spring Boot 프로젝트에서 비즈니스 로직을 구현하고 관리하는 공간입니다. 이 패키지는 주로 애플리케이션의 핵심 기능을 정의하는 클래스들을 포함하며, 데이터 처리, 트랜잭션 관리, 유효성 검사 등과 같은 복잡한 로직을 수행합니다.<br>
예를 들어, UserService 클래스는 사용자 등록, 로그인, 정보 수정 등의 기능을 처리하는 비즈니스 로직을 포함할 수 있습니다. 이 클래스는 repository 패키지에 정의된 데이터 접근 객체를 호출하여 데이터를 조회하거나 저장하고, 필요한 경우 추가적인 비즈니스 규칙을 적용하여 결과를 반환합니다.<br>
service 패키지를 사용하면 비즈니스 로직과 데이터 접근 로직을 분리하여 코드의 가독성과 유지보수성을 높일 수 있습니다. 이렇게 하면 개발자는 각 기능의 구현을 독립적으로 테스트하고 관리할 수 있으며, 향후 비즈니스 요구사항의 변경이나 기능 추가에 유연하게 대응할 수 있습니다.<br>
결론적으로, service 패키지는 비즈니스 로직을 구현하고 관리하는 공간으로, 애플리케이션의 핵심 기능을 효율적으로 처리하고, 코드의 유지보수성과 확장성을 높이는 중요한 역할을 수행합니다.

service는 이름 그대로 이 서비스의 중요로직을 구현하고 관리하는 패키지이다. 주로 컨트롤러와 데이터 레포지토리 간의 중재 역할을 한다
